# v0.3.0: HNSW, Prometheus, Per-Tenant Metrics

## Overview
Performance and observability upgrade for StackIntercept. Replace linear Vec scan with HNSW
(speeding up semantic retrieval above ~10k entries), expose Prometheus metrics, and break down
counters per-tenant for multi-tenant deployments.

## Changes

### 1. HNSW Semantic Indexing
**File:** `src/cache.rs`

Replace the linear `Vec<CacheItem>` scan with a HNSW index inside each semantic bucket.

**Architecture:**
- `DashMap<String, SemanticBucket>` where `SemanticBucket` contains:
  - `hnsw: HnswIndex` — the HNSW index (stores entry IDs with 384-dim vectors)
  - `items: Vec<Option<CacheItem>>` — flat array indexed by HNSW entry ID
  - A free-list for reclaimed IDs on eviction
- On insert: encode vector into HNSW, store `CacheItem` at the returned slot
- On lookup: query HNSW for nearest neighbors (top-1 with threshold 0.93), check cosine
  directly for verification
- On eviction: remove entry from HNSW + mark slot as `None` in items vec

**Why defer from 0.2.x:**
At 0–10k entries, linear scan is faster than HNSW build overhead. HNSW becomes worth it
at larger cache sizes. We keep `Vec+cosine` as the default, with an opt-in `hnsw` feature
for high-volume deployments.

**Dependency:** `fast-hnsw` already in `Cargo.toml`.

**Key design decisions:**
- HNSW remains per-bucket (isolated by context key). No global HNSW.
- Cosine verification on HNSW results (HNSW returns approximate nearest neighbors; we still
  verify the 0.93 threshold via raw dot product).
- Fallback to linear scan if HNSW build fails or bucket is small (< threshold).

### 2. Semantic Bucket Eviction by Item ID
**File:** `src/cache.rs`

Currently `evict_global()` removes items from the largest bucket. Replace with an API that
evicts a specific item by ID within a bucket.

**API:**
```rust
pub fn evict_item(bucket_id: &str, item_id: usize) -> bool;
```

The `item_id` is returned by the semantic insert path. The admin route
`DELETE /admin/cache/semantic/:context_key` can optionally accept an `item_id` query
parameter to delete individual items rather than entire buckets.

### 3. Per-Tenant Metrics
**File:** `src/main.rs`

Replace the flat `Metrics` struct with tenant-scoped counters.

**Architecture:**
- `Metrics` struct keeps global totals AND a `DashMap<String, TenantMetrics>`
- `TenantMetrics` has the same 8 atomic counters as the global `Metrics`
- At each cache/route decision point, increment both the global AND the tenant counter
- Tenant key is `"_default_"` when no tenant header is configured
- Admin metrics endpoint returns both global and per-tenant breakdown

**Response format:**
```json
{
  "uptime_secs": 3600,
  "global": {
    "exact_hits": 142,
    "semantic_hits": 23,
    "misses": 89,
    ...
  },
  "tenants": {
    "tenant-alpha": { "exact_hits": 100, ... },
    "tenant-beta": { "exact_hits": 42, ... }
  }
}
```

### 4. Prometheus Endpoint Format
**File:** `src/main.rs`

Add `GET /admin/metrics/prometheus` returning OpenMetrics-compatible text format.

**Format:**
```
# HELP stack_intercept_exact_hits Total exact cache hits
# TYPE stack_intercept_exact_hits counter
stack_intercept_exact_hits{tenant="alpha"} 100
stack_intercept_exact_hits{tenant="beta"} 42
stack_intercept_semantic_hits_total{tenant="alpha"} 23
...
```

**Implementation:**
- Iterate all metrics (global + per-tenant) and format as Prometheus exposition text
- Content-type: `text/plain; version=0.0.4`
- No external Prometheus client library — format directly as a string to keep deps minimal

### 5. Per-Request Opt-Out Header for Semantic Cache
**File:** `src/main.rs`

Allow clients to opt out of semantic caching per-request via
`x-stack-intercept-no-semantic-cache: true` header. Exact cache still applies.
Useful for requests with sensitive or non-repetitive content.

## Config changes

| Env Var | Default | Description |
|---|---|---|
| `STACK_INTERCEPT_USE_HNSW` | `false` | Enable HNSW indexing (opt-in) |
| `STACK_INTERCEPT_HNSW_EF_CONSTRUCTION` | `200` | HNSW construction quality |
| `STACK_INTERCEPT_HNSW_M` | `16` | HNSW max connections per layer |

## Migration path (from v0.2.x)

- All existing config/env vars remain unchanged
- HNSW is opt-in (default: off, linear scan)
- Tenant metrics are additive (new fields in existing response)
- Prometheus is a new endpoint (no change to existing `/admin/metrics`)
- Semantic cache item eviction is a new optional query param on existing route

## Test plan
1. Unit tests for HNSW insert/lookup/eviction within a bucket
2. Unit tests for `evict_item()` by ID
3. Integration tests for `/admin/metrics/prometheus` endpoint
4. Integration tests for per-tenant metric counters
5. HNSW correctness test: verify HNSW results match linear scan results for same vectors
6. Backward-compatibility: existing semantic mode continues working without HNSW enabled

## Rough implementation order
1. Per-tenant metrics (lowest risk, purely additive)
2. Prometheus endpoint (depends on per-tenant metrics for full value)
3. Semantic eviction by item ID (small, focused change)
4. HNSW indexing (largest risk, needs careful testing)
5. Per-request sem cache opt-out header (trivial)
